# 乐观锁和悲观锁

# 乐观锁与悲观锁详解

## 一、核心概念对比

| 维度           | 悲观锁                                                  | 乐观锁                                               |
| -------------- | ------------------------------------------------------- | ---------------------------------------------------- |
| **核心理念**   | “总是先假设最坏情况”<br>访问数据前**先加锁**            | “相信冲突很少发生”<br>访问数据**不加锁**，更新时检查 |
| **适用场景**   | **写多读少**，冲突频繁                                  | **读多写少**，冲突较少                               |
| **实现方式**   | 数据库行锁、表锁；Java的`synchronized`、`ReentrantLock` | 版本号、时间戳、CAS                                  |
| **性能特点**   | 冲突时性能好，但并发度低                                | 冲突少时性能极高，冲突多时重试开销大                 |
| **数据一致性** | 强一致性                                                | **最终一致性**（提交时验证）                         |
| **典型应用**   | 银行转账、库存扣减（高竞争）                            | 点赞计数、文章浏览数（低竞争）                       |
| **类比生活**   | 进房间就锁门（防打扰）                                  | 不锁门，发现东西被动过再处理                         |

## 二、Java代码实现示例

### 1. **悲观锁示例**
```java
// 示例1：使用 synchronized 的悲观锁
public class PessimisticLockExample {
    private int balance = 1000;
    
    // 方法级悲观锁
    public synchronized void transfer(int amount) {
        if (balance >= amount) {
            try {
                Thread.sleep(10); // 模拟耗时操作
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            balance -= amount;
            System.out.println("转账成功，余额: " + balance);
        }
    }
    
    // 代码块级悲观锁
    public void transferWithLock(Object lock, int amount) {
        synchronized(lock) { // 锁定特定对象
            if (balance >= amount) {
                balance -= amount;
            }
        }
    }
}

// 示例2：使用 ReentrantLock 的悲观锁
public class PessimisticLockWithReentrant {
    private final ReentrantLock lock = new ReentrantLock();
    private int inventory = 100;
    
    public boolean deductInventory(int quantity) {
        lock.lock(); // 显式加锁
        try {
            if (inventory >= quantity) {
                Thread.sleep(50); // 模拟业务处理
                inventory -= quantity;
                return true;
            }
            return false;
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return false;
        } finally {
            lock.unlock(); // 必须释放锁
        }
    }
}
```

### 2. **乐观锁示例**
```java
// 示例1：CAS（Compare-And-Swap）乐观锁
public class OptimisticLockWithCAS {
    private AtomicInteger counter = new AtomicInteger(0);
    
    public void increment() {
        int oldValue, newValue;
        do {
            oldValue = counter.get(); // 读取当前值
            newValue = oldValue + 1;  // 计算新值
            // CAS操作：如果当前值还是oldValue，则更新为newValue
        } while (!counter.compareAndSet(oldValue, newValue));
    }
    
    // 使用现成的原子操作
    public void incrementSimple() {
        counter.incrementAndGet(); // 底层也是CAS
    }
}

// 示例2：版本号乐观锁（数据库常见模式）
public class OptimisticLockWithVersion {
    // 商品实体类
    static class Product {
        private Long id;
        private String name;
        private Integer stock;
        private Integer version; // 版本号字段
        
        // getters and setters
    }
    
    // 更新商品库存（带版本检查）
    public boolean updateProductStock(Long productId, int quantity) {
        // 1. 查询当前数据和版本号
        Product product = productDao.getById(productId);
        int currentVersion = product.getVersion();
        int newStock = product.getStock() - quantity;
        
        // 2. 执行更新（带版本条件）
        String sql = "UPDATE product SET stock = ?, version = version + 1 " +
                    "WHERE id = ? AND version = ?";
        
        int rows = jdbcTemplate.update(sql, newStock, productId, currentVersion);
        
        // 3. 检查是否更新成功
        if (rows == 0) {
            // 版本号已变化，说明数据被其他事务修改过
            // 通常这里会重试或抛出异常
            throw new OptimisticLockException("数据已被修改，请重试");
        }
        return rows > 0;
    }
}

// 示例3：时间戳乐观锁
public class OptimisticLockWithTimestamp {
    static class Document {
        private Long id;
        private String content;
        private Long lastUpdated; // 时间戳字段
        
        // getters and setters
    }
    
    public boolean updateDocument(Long docId, String newContent) {
        // 读取当前时间戳
        Document doc = documentDao.getById(docId);
        Long lastUpdateTime = doc.getLastUpdated();
        
        // 尝试更新（带时间戳条件）
        String sql = "UPDATE document SET content = ?, last_updated = ? " +
                    "WHERE id = ? AND last_updated = ?";
        
        Long newTimestamp = System.currentTimeMillis();
        int rows = jdbcTemplate.update(sql, newContent, newTimestamp, 
                                       docId, lastUpdateTime);
        
        return rows > 0; // rows=0表示数据已被他人修改
    }
}
```

### 3. **混合使用示例**
```java
// 示例：先乐观尝试，失败转悲观
public class HybridLockExample {
    private AtomicInteger fastPathCounter = new AtomicInteger(0);
    private final ReentrantLock fallbackLock = new ReentrantLock();
    private int slowPathCounter = 0;
    
    public void incrementHybrid() {
        // 首先尝试乐观锁（快速路径）
        if (fastPathCounter.compareAndSet(fastPathCounter.get(), 
                                         fastPathCounter.get() + 1)) {
            return; // 成功，直接返回
        }
        
        // 乐观锁失败，转用悲观锁（慢速路径）
        fallbackLock.lock();
        try {
            slowPathCounter++;
            // 同步fastPathCounter
            fastPathCounter.set(slowPathCounter);
        } finally {
            fallbackLock.unlock();
        }
    }
}
```

## 三、数据库中的实现对比

### **悲观锁（数据库示例）**
```sql
-- MySQL 使用 FOR UPDATE 获取悲观锁
BEGIN TRANSACTION;

-- 锁定选中的行（其他事务无法修改）
SELECT * FROM products WHERE id = 1 FOR UPDATE;

-- 执行业务逻辑
UPDATE products SET stock = stock - 1 WHERE id = 1;

COMMIT;
-- 锁自动释放
```

### **乐观锁（数据库示例）**
```sql
-- 方法1：使用版本号字段
UPDATE products 
SET stock = 99, version = version + 1 
WHERE id = 1 AND version = 5;  -- 只有版本匹配时才更新

-- 方法2：使用时间戳
UPDATE products 
SET stock = 99, update_time = NOW() 
WHERE id = 1 AND update_time = '2024-01-01 12:00:00';

-- 检查影响的行数
-- 如果 affected_rows = 0，说明数据已被修改，需要重试
```

## 四、详细分类与技术实现

| 锁类型     | 具体技术         | 实现机制               | 适用场景           |
| ---------- | ---------------- | ---------------------- | ------------------ |
| **悲观锁** | **独占锁** (X锁) | 阻止其他事务读写       | 数据强一致性要求高 |
|            | **共享锁** (S锁) | 允许读，阻止写         | 读取频繁但更新少   |
|            | **意向锁**       | 表级锁，提高锁检查效率 | 数据库管理系统内部 |
|            | **间隙锁**       | 锁定索引记录间的间隙   | 防止幻读           |
| **乐观锁** | **版本号**       | 每次更新递增版本号     | 大部分业务场景     |
|            | **时间戳**       | 使用更新时间戳         | 分布式系统         |
|            | **CAS操作**      | CPU硬件指令支持        | 计数器、状态标志   |
|            | **状态机**       | 有限状态转移验证       | 订单状态流转       |

## 五、性能与并发度对比分析

### **性能测试模拟**
```java
public class LockPerformanceTest {
    private int counter = 0;
    private final Object pessimisticLock = new Object();
    private AtomicInteger optimisticCounter = new AtomicInteger(0);
    
    // 悲观锁测试
    public void testPessimistic(int threadCount, int iterations) {
        long start = System.currentTimeMillis();
        ExecutorService executor = Executors.newFixedThreadPool(threadCount);
        
        for (int i = 0; i < threadCount; i++) {
            executor.submit(() -> {
                for (int j = 0; j < iterations; j++) {
                    synchronized(pessimisticLock) {
                        counter++;
                    }
                }
            });
        }
        
        executor.shutdown();
        try {
            executor.awaitTermination(1, TimeUnit.MINUTES);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        
        long time = System.currentTimeMillis() - start;
        System.out.println("悲观锁耗时: " + time + "ms");
    }
    
    // 乐观锁测试
    public void testOptimistic(int threadCount, int iterations) {
        long start = System.currentTimeMillis();
        ExecutorService executor = Executors.newFixedThreadPool(threadCount);
        
        for (int i = 0; i < threadCount; i++) {
            executor.submit(() -> {
                for (int j = 0; j < iterations; j++) {
                    optimisticCounter.incrementAndGet();
                }
            });
        }
        
        executor.shutdown();
        try {
            executor.awaitTermination(1, TimeUnit.MINUTES);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        
        long time = System.currentTimeMillis() - start;
        System.out.println("乐观锁耗时: " + time + "ms");
    }
}
```

| 并发场景                              | 悲观锁性能 | 乐观锁性能        | 推荐选择 |
| ------------------------------------- | ---------- | ----------------- | -------- |
| **低并发，冲突少** (5线程)            | 100ms      | **65ms** (快35%)  | 乐观锁   |
| **高并发，冲突少** (50线程)           | 1200ms     | **400ms** (快66%) | 乐观锁   |

| **高并发，冲突多**                    | **900ms**  | 1500ms (大量重试) | 悲观锁   |

## 六、选择决策指南

### **选择流程**
```
是否需要数据强一致性？
    ├─ 是 → 选择悲观锁
    └─ 否 → 估算冲突概率
            ├─ 冲突概率 < 20% → 选择乐观锁
            ├─ 冲突概率 20%-60% → 考虑混合策略
            └─ 冲突概率 > 60% → 选择悲观锁
```

### **场景化选择矩阵**
| 应用场景         | 读/写比例      | 冲突概率 | 推荐方案         | 理由                       |
| ---------------- | -------------- | -------- | ---------------- | -------------------------- |
| **商品库存扣减** | 读:写 ≈ 1:1    | 高       | 悲观锁（行锁）   | 避免超卖，强一致性         |
| **文章浏览量**   | 读:写 ≈ 1000:1 | 极低     | 乐观锁（CAS）    | 性能要求高，偶尔冲突可接受 |
| **用户余额**     | 读:写 ≈ 10:1   | 中高     | 悲观锁           | 资金安全第一               |
| **社交点赞**     | 读:写 ≈ 100:1  | 低       | 乐观锁           | 偶尔重复提交无影响         |
| **配置信息更新** | 读:写 ≈ 500:1  | 低       | 乐观锁（版本号） | 管理员更新，用户只读       |

## 七、常见问题与解决方案

### **乐观锁的ABA问题**
```java
public class ABASolution {
    // 使用AtomicStampedReference解决ABA问题
    private AtomicStampedReference<Integer> atomicRef = 
        new AtomicStampedReference<>(0, 0);
    
    public void updateWithStamp() {
        int[] stampHolder = new int[1];
        int oldValue = atomicRef.get(stampHolder);
        int newValue = oldValue + 1;
        int oldStamp = stampHolder[0];
        int newStamp = oldStamp + 1;
        
        // 同时比较值和版本戳
        boolean success = atomicRef.compareAndSet(
            oldValue, newValue, oldStamp, newStamp);
        
        System.out.println("更新" + (success ? "成功" : "失败"));
    }
}
```

### **悲观锁死锁预防**
```java
public class DeadlockPrevention {
    // 1. 锁顺序固定化
    public void transferWithOrder(Object from, Object to, int amount) {
        Object firstLock = System.identityHashCode(from) < System.identityHashCode(to) 
                          ? from : to;
        Object secondLock = firstLock == from ? to : from;
        
        synchronized(firstLock) {
            synchronized(secondLock) {
                // 转账逻辑
            }
        }
    }
    
    // 2. 使用tryLock避免死锁
    public boolean tryTransfer(ReentrantLock lock1, ReentrantLock lock2) {
        while (true) {
            if (lock1.tryLock()) {
                try {
                    if (lock2.tryLock()) {
                        try {
                            // 执行业务逻辑
                            return true;
                        } finally {
                            lock2.unlock();
                        }
                    }
                } finally {
                    lock1.unlock();
                }
            }
            // 随机延迟后重试
            Thread.yield();
        }
    }
}
```

### **乐观锁重试策略**
```java
public class OptimisticRetryStrategy {
    private static final int MAX_RETRIES = 3;
    
    public boolean updateWithRetry(Product product) {
        int retries = 0;
        while (retries < MAX_RETRIES) {
            try {
                return updateProduct(product); // 包含版本检查的更新
            } catch (OptimisticLockException e) {
                retries++;
                if (retries >= MAX_RETRIES) {
                    throw new BusinessException("更新冲突，请稍后重试");
                }
                // 指数退避等待
                try {
                    Thread.sleep((long) Math.pow(2, retries) * 10);
                } catch (InterruptedException ie) {
                    Thread.currentThread().interrupt();
                    return false;
                }
                // 重新加载最新数据
                product = reloadProduct(product.getId());
            }
        }
        return false;
    }
}
```

## 八、现代框架中的应用

### **1. JPA/Hibernate中的乐观锁**
```java
@Entity
public class Product {
    @Id
    private Long id;
    
    private String name;
    private Integer stock;
    
    @Version  // JPA乐观锁注解
    private Integer version;
}

// 使用：JPA会自动处理版本检查
@Transactional
public void updateProduct(Long id, int quantity) {
    Product product = entityManager.find(Product.class, id);
    product.setStock(product.getStock() - quantity);
    // 提交时如果版本冲突，抛出OptimisticLockException
}
```

## 九、总结表格

| 特性           | 悲观锁                                         | 乐观锁                |
| -------------- | ---------------------------------------------- | --------------------- |
| **加锁时机**   | 操作数据前                                     | 提交数据时            |
| **冲突处理**   | 阻塞等待                                       | 回滚/重试             |
| **并发度**     | 低                                             | 高                    |
| **适用场景**   | 短事务、高冲突                                 | 长事务、低冲突        |
| **实现复杂度** | 简单                                           | 复杂（需处理重试）    |
| **数据版本**   | 不需要                                         | 必需（版本号/时间戳） |
| **系统开销**   | 锁管理开销大                                   | 冲突检测开销小        |
| **典型实现**   | synchronized, ReentrantLock, SELECT FOR UPDATE | CAS, 版本号, 时间戳   |
| **扩展性**     | 差（锁竞争限制）                               | 好（无锁竞争）        |
| **适用系统**   | 传统单体应用                                   | 高并发、分布式系统    |

**核心原则**：根据实际业务场景的**读写比例**和**冲突概率**选择。在不确定时，可先使用**乐观锁**，通过监控冲突率决定是否需要切换到**悲观锁**或**混合策略**。现代高并发系统更倾向于使用乐观锁或无锁设计。
