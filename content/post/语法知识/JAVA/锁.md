# 锁

# Java 锁机制全面总结

## 一、锁的类型分类

### 1. **按锁的性质分类**

| 类型                 | 实现                               | 特点                                 | 适用场景                 |
| -------------------- | ---------------------------------- | ------------------------------------ | ------------------------ |
| **悲观锁**           | synchronized, ReentrantLock        | 默认认为会发生并发冲突，先加锁再操作 | 写多读少，竞争激烈       |
| **乐观锁**           | CAS, Atomic类, 版本号              | 默认认为不会冲突，先操作再验证       | 读多写少，竞争不激烈     |
| **可重入锁**         | synchronized, ReentrantLock        | 同一个线程可重复获取同一把锁         | 递归调用，需要重入的场景 |
| **不可重入锁**       | 自定义实现                         | 不可重复获取，会死锁                 | 特殊需求场景             |
| **公平锁**           | ReentrantLock(true)                | 按等待顺序获取锁                     | 需要避免线程饥饿         |
| **非公平锁**         | synchronized, ReentrantLock(false) | 可插队获取锁，性能更高               | 追求吞吐量               |
| **独占锁（排他锁）** | synchronized, ReentrantLock        | 同一时刻只允许一个线程访问           | 写操作，数据修改         |
| **共享锁**           | ReentrantReadWriteLock.ReadLock    | 允许多个线程同时访问                 | 读操作，数据查询         |
| **读写锁**           | ReentrantReadWriteLock             | 读共享，写独占                       | 读多写少，如缓存         |

### 2. **按锁的实现机制分类**

| 类型         | 实现                        | 特点                      | 原理             |
| ------------ | --------------------------- | ------------------------- | ---------------- |
| **自旋锁**   | AtomicReference, CAS        | 循环尝试获取锁，不放弃CPU | CAS + 循环       |
| **互斥锁**   | synchronized, ReentrantLock | 获取失败则阻塞线程        | 操作系统内核调度 |
| **分段锁**   | ConcurrentHashMap           | 将锁细粒度化，减少竞争    | 数组 + 锁分段    |
| **偏向锁**   | synchronized (JVM优化)      | 无竞争时消除同步开销      | 对象头Mark Word  |
| **轻量级锁** | synchronized (JVM优化)      | 短时间竞争时自旋          | CAS + 自旋       |
| **重量级锁** | synchronized (JVM优化)      | 竞争激烈时阻塞            | 操作系统互斥量   |
| **邮戳锁**   | StampedLock (JDK8+)         | 支持乐观读，避免写饥饿    | 戳记验证机制     |

## 二、具体锁的使用

### 1. **synchronized（内置锁/监视器锁）**

```
// 1. 同步代码块
public void method() {
    synchronized (this) {  // 对象锁
        // 临界区代码
    }
}

// 2. 同步实例方法
public synchronized void instanceMethod() {
    // 锁是当前实例对象
}

// 3. 同步静态方法
public static synchronized void staticMethod() {
    // 锁是当前类的Class对象
}
```

**特点**：

- 

  自动释放锁

- 

  不可中断（除非抛出异常）

- 

  非公平锁

- 

  可重入

- 

  JVM内置支持

### 2. **ReentrantLock（可重入锁）**

```
private final ReentrantLock lock = new ReentrantLock();

public void method() {
    lock.lock();  // 获取锁
    try {
        // 临界区代码
    } finally {
        lock.unlock();  // 必须手动释放
    }
}
```

**高级特性**：

```
// 1. 尝试获取锁（非阻塞）
if (lock.tryLock()) {
    try {
        // 获取成功
    } finally {
        lock.unlock();
    }
}

// 2. 带超时的尝试获取锁
if (lock.tryLock(1, TimeUnit.SECONDS)) {
    try {
        // 获取成功
    } finally {
        lock.unlock();
    }
}

// 3. 可中断的获取锁
try {
    lock.lockInterruptibly();
    try {
        // 临界区
    } finally {
        lock.unlock();
    }
} catch (InterruptedException e) {
    // 处理中断
}

// 4. 公平锁
ReentrantLock fairLock = new ReentrantLock(true);

// 5. 条件变量
Condition condition = lock.newCondition();
condition.await();      // 等待
condition.signal();     // 唤醒一个
condition.signalAll();  // 唤醒所有
```

### 3. **ReentrantReadWriteLock（读写锁）**

```
private final ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
private final Lock readLock = rwLock.readLock();
private final Lock writeLock = rwLock.writeLock();

// 读操作
public Object read() {
    readLock.lock();
    try {
        return data;
    } finally {
        readLock.unlock();
    }
}

// 写操作
public void write(Object value) {
    writeLock.lock();
    try {
        data = value;
    } finally {
        writeLock.unlock();
    }
}
```

**特性**：

- 

  读锁共享，写锁独占

- 

  写锁可降级为读锁，但读锁不能升级为写锁

- 

  支持公平/非公平模式

- 

  可重入

### 4. **StampedLock（邮戳锁）**

```
private final StampedLock stampedLock = new StampedLock();
private double x, y;

// 1. 乐观读
public double distanceFromOrigin() {
    long stamp = stampedLock.tryOptimisticRead();  // 乐观读
    double currentX = x, currentY = y;
    if (!stampedLock.validate(stamp)) {  // 检查数据是否被修改
        stamp = stampedLock.readLock();  // 升级为悲观读锁
        try {
            currentX = x;
            currentY = y;
        } finally {
            stampedLock.unlockRead(stamp);
        }
    }
    return Math.sqrt(currentX * currentX + currentY * currentY);
}

// 2. 悲观读
public void move(double deltaX, double deltaY) {
    long stamp = stampedLock.writeLock();
    try {
        x += deltaX;
        y += deltaY;
    } finally {
        stampedLock.unlockWrite(stamp);
    }
}
```

**特点**：

- 

  支持乐观读，提高读性能

- 

  写锁不可重入

- 

  不支持条件变量

- 

  适合读多写少的场景

## 三、原子类与CAS

### 1. **Atomic类（基于CAS的乐观锁）**

```
// 常用原子类
AtomicInteger count = new AtomicInteger(0);
AtomicLong longValue = new AtomicLong(0L);
AtomicBoolean flag = new AtomicBoolean(false);
AtomicReference<String> reference = new AtomicReference<>("init");

// 原子操作
count.incrementAndGet();        // i++
count.getAndIncrement();        // ++i
count.addAndGet(5);             // i += 5
count.compareAndSet(expect, update);  // CAS
```

### 2. **CAS操作原理**

```
// CAS伪实现
public boolean compareAndSet(int expect, int update) {
    int current = getValue();
    if (current == expect) {
        setValue(update);
        return true;
    }
    return false;
}
```

**CAS问题**：

- 

  **ABA问题**：通过AtomicStampedReference/AtomicMarkableReference解决

- 

  **循环时间长开销大**：自旋消耗CPU

- 

  **只能保证一个变量的原子操作**：多个变量需用锁

## 四、锁优化技术

### 1. **JVM对synchronized的优化**

| 锁状态   | 存储内容                 | 标记位 | 说明                   |
| -------- | ------------------------ | ------ | ---------------------- |
| 无锁     | 对象的hashCode、分代年龄 | 01     | 对象未锁定             |
| 偏向锁   | 线程ID、Epoch、分代年龄  | 01     | 适用于只有一个线程访问 |
| 轻量级锁 | 指向栈中锁记录的指针     | 00     | 短时间竞争，自旋等待   |
| 重量级锁 | 指向互斥量的指针         | 10     | 竞争激烈，线程阻塞     |
| GC标记   | 空                       | 11     | GC标记，与锁无关       |

**锁升级流程**：

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁
```

### 2. **锁消除**

```
// 逃逸分析后，sb未逃逸，锁可消除
public String concat(String s1, String s2, String s3) {
    StringBuffer sb = new StringBuffer();
    sb.append(s1);
    sb.append(s2);
    sb.append(s3);
    return sb.toString();  // sb在方法内部创建和使用，不会逃逸
}
```

### 3. **锁粗化**

```
// JVM优化前
for (int i = 0; i < 100; i++) {
    synchronized (this) {  // 频繁加锁解锁
        doSomething();
    }
}

// JVM优化后（锁粗化）
synchronized (this) {
    for (int i = 0; i < 100; i++) {
        doSomething();
    }
}
```

## 五、锁的使用场景

### 1. **并发容器使用锁的类型**

| 容器                  | 锁类型          | 特点                                  |
| --------------------- | --------------- | ------------------------------------- |
| ConcurrentHashMap     | 分段锁+CAS      | JDK7: 分段锁; JDK8+: synchronized+CAS |
| CopyOnWriteArrayList  | 写时复制锁      | 写操作加锁，读操作无锁                |
| ConcurrentLinkedQueue | CAS             | 无锁队列                              |
| ArrayBlockingQueue    | ReentrantLock   | 阻塞队列                              |
| LinkedBlockingQueue   | 双ReentrantLock | 分离读写锁                            |
| PriorityBlockingQueue | ReentrantLock   | 优先队列                              |

### 2. **场景选择决策表**

| 场景特征                       | 推荐锁类型                 | 理由                   |
| ------------------------------ | -------------------------- | ---------------------- |
| 简单同步，代码简洁             | synchronized               | 自动释放，无需手动管理 |
| 需要高级功能（可中断、超时等） | ReentrantLock              | 功能丰富，灵活控制     |
| 读多写少，读性能要求高         | ReentrantReadWriteLock     | 读写分离，提高并发     |
| 读多写少，且读不经常与写冲突   | StampedLock                | 乐观读提升性能         |
| 单个变量的原子操作             | Atomic类                   | 无锁，性能高           |
| 多个变量的复合操作             | synchronized/ReentrantLock | 保证原子性             |
| 线程池任务执行                 | 无锁或CAS                  | 减少上下文切换         |
| 缓存更新                       | ReadWriteLock              | 读共享，写独占         |

## 六、死锁与解决方案

### 1. **死锁产生条件（必要条件）**

1. 

   互斥条件

2. 

   请求与保持条件

3. 

   不剥夺条件

4. 

   循环等待条件

### 2. **死锁示例**

```
// 死锁示例
public class DeadLockExample {
    private final Object lock1 = new Object();
    private final Object lock2 = new Object();
    
    public void method1() {
        synchronized (lock1) {
            synchronized (lock2) {  // 可能死锁
                // ...
            }
        }
    }
    
    public void method2() {
        synchronized (lock2) {
            synchronized (lock1) {  // 可能死锁
                // ...
            }
        }
    }
}
```

### 3. **死锁预防**

| 策略       | 方法                | 示例                              |
| ---------- | ------------------- | --------------------------------- |
| 顺序加锁   | 统一获取锁的顺序    | 都先获取lock1，再获取lock2        |
| 超时放弃   | tryLock(timeout)    | lock.tryLock(5, TimeUnit.SECONDS) |
| 可中断锁   | lockInterruptibly() | 响应中断请求                      |
| 银行家算法 | 系统资源分配        | 避免进入不安全状态                |

## 七、性能对比

### 1. **性能比较表**

| 场景     | synchronized | ReentrantLock | ReadWriteLock | StampedLock乐观读 |
| -------- | ------------ | ------------- | ------------- | ----------------- |
| 高竞争写 | 中           | 高            | 低            | 低                |
| 高竞争读 | 低           | 低            | 高            | 高                |
| 低竞争   | 高           | 中            | 中            | 高                |
| 内存占用 | 低           | 中            | 高            | 中                |

### 2. **选择建议**

1. 

   **优先使用synchronized**：简单场景，JVM优化好

2. 

   **需要高级功能用ReentrantLock**：可中断、超时、公平锁等

3. 

   **读多写少用ReadWriteLock或StampedLock**

4. 

   **单变量操作用Atomic类**

5. 

   **避免过度优化**：过早优化是万恶之源

## 八、最佳实践

### 1. **锁使用规范**

```
// 1. 锁对象声明为final
private final Object lock = new Object();

// 2. 缩小锁范围
public void method() {
    // 非临界区代码
    synchronized (lock) {
        // 临界区代码（尽量简短）
    }
    // 非临界区代码
}

// 3. 避免锁嵌套
// 4. 使用tryLock避免死锁
// 5. 清理资源在finally块
```

### 2. **避免的常见问题**

| 问题       | 错误示例               | 正确做法              |
| ---------- | ---------------------- | --------------------- |
| 锁泄露     | 未在finally中释放锁    | 确保finally中释放锁   |
| 过度同步   | 锁范围过大             | 缩小同步范围          |
| 锁对象不当 | 使用String等常量作为锁 | 使用private final对象 |
| 死锁       | 嵌套获取多把锁         | 按顺序获取锁          |

## 十、总结要点

1. 

   **理解锁的层次**：从乐观锁到重量级锁，根据场景选择

2. 

   **明确需求**：是否需要公平、可中断、超时等特性

3. 

   **优先简单**：synchronized能满足就不用复杂方案

4. 

   **注意性能**：锁竞争是性能瓶颈，尽量减少锁粒度

5. 

   **避免死锁**：按顺序获取锁，使用tryLock

6. 

   **监控诊断**：利用工具定位锁问题

7. 

   **与时俱进**：了解新版本JDK的锁优化（如JDK15+的偏向锁禁用等）
